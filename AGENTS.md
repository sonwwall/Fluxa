# AGENTS.md

## 先读文档

开发前先阅读与任务相关的文档，不要凭通用经验自行决定架构。

- 前端任务读 `docs/前端开发规范.md`
- 后端任务读 `docs/后端开发规范.md`
- 需求和范围不清楚时读 `docs/前期开发准备文档.md`

前后端的具体技术选型、目录结构、分层规则、测试要求以这些文档为准

## 工作边界

- 只修改和当前任务直接相关的文件。
- 不要顺手重构、改名、搬目录、统一格式或清理无关代码。
- 不要提前实现未确认功能；准备文档中标记为待确认的内容，必须先询问。
- 新增依赖、调整目录结构、修改接口格式、改变权限逻辑前，先确认是否符合对应开发规范。
- 如果发现现有代码和规范冲突，先说明冲突点，再选择最小改动方案。
- 修改文件后默认不要提交 commit，也不要询问是否提交。
- 只有用户明确要求 `commit` 时，才读取 git 变动区并提交一次。
- commit 信息遵循 Conventional Commits，格式为 `<type>(scope): short description`，例如 `feat(auth): add login endpoint`。
- `type` 按实际变更选择，例如 `docs`、`feat`、`fix`、`refactor`、`test`、`chore`等。
- commit 正文使用英文，并按条目分点描述具体修改。

## 敏感文件和配置

- 不要随意修改 `.env`、`.env.*`、配置模板、部署脚本、CI 配置、数据库迁移文件。
- 必须修改配置时，说明原因、影响范围和需要的环境变量。
- 不要提交密钥、Token、密码、Cookie、JWT Secret、数据库连接串或真实用户数据。
- 不要把本地机器路径、临时调试地址、个人账号信息写入项目文件。
- 不要为了让代码跑通而降低鉴权、关闭校验或绕过错误处理。

## 代码习惯

- 优先沿用已有命名、目录、分层和错误处理方式。
- 新代码应放在最接近其业务归属的位置，只有确实复用时才提升为共享代码。
- 注释只解释业务约束、边界条件和不明显的决策，不复述代码行为。
- 删除代码前确认没有被路由、测试、脚本或文档引用。
- 修改公开接口、响应结构、错误码或配置项时，同步更新相关文档。

## 验证要求

完成后根据改动范围运行必要检查。

- 前端改动优先运行类型检查、Lint、测试和构建。
- 后端改动优先运行相关单元测试或 `go test ./...`。
- UI 改动需要检查桌面端和移动端的关键页面状态。
- 如果检查无法运行，说明具体原因，不要假装已验证。

## 回复要求

交付时简要说明：

- 改了哪些文件。
- 解决了什么问题。
- 做了哪些验证。
- 还有哪些风险或未完成项，如没有可以不用说。


## CI/CD 规则说明

本项目使用自建 GitLab CI/CD，不使用 GitHub Actions。CI 配置文件为仓库根目录下的 `.gitlab-ci.yml`。当前 GitLab Runner 已注册并在线，Runner 标签为 `debian`，因此所有需要由该 Runner 执行的 job 都必须显式声明：

```yaml
tags:
  - debian
```

项目主要维护两个分支：`dev` 和 `main`。编写 CI/CD 时应按分支区分行为：

* `dev` 分支用于日常开发、测试和测试环境部署，可以自动执行测试、构建和 dev 环境部署。
* `main` 分支用于稳定版本和生产发布，必须更加保守。生产部署类 job 应使用 `when: manual`，避免 push 后自动部署到生产环境。
* 推荐使用 `rules` 控制分支行为，不建议使用已过时或不够灵活的 `only/except`。
* 判断当前分支时使用 GitLab 内置变量 `$CI_COMMIT_BRANCH`。

示例：

```yaml
rules:
  - if: '$CI_COMMIT_BRANCH == "dev"'
```

或：

```yaml
rules:
  - if: '$CI_COMMIT_BRANCH == "main"'
```

CI 配置应优先保持清晰、可读、可维护。对于 dev 和 main 行为不同的流程，优先拆分为独立 job，例如 `deploy-dev` 和 `deploy-prod`，不要把大量分支判断塞进同一个脚本里。

除非明确要求，不要修改 GitLab Runner 注册方式、Runner 标签、远程仓库地址、Docker 端口映射或 GitLab 部署配置。

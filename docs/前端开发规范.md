# 前端开发规范

本文档是 Fluxa 前端开发的执行基线。Agent 开发前必须阅读，并优先遵循现有代码与本规范。

## 1. 技术基线

- 使用 Next.js App Router、React、TypeScript、HeroUI、Tailwind CSS、Shiki。
- TypeScript 开启严格模式，禁止无理由使用 `any`。
- 第一阶段仅支持 Markdown；未经确认不得引入 MDX、新框架、状态库或 UI 库。
- 包管理器统一使用 `pnpm`，禁止使用 npm、Yarn 或 Bun 安装依赖及运行脚本。
- 仅提交 `pnpm-lock.yaml`；不得提交其他包管理器生成的锁文件。
- 添加、删除或升级依赖时必须同步更新 `package.json` 与 `pnpm-lock.yaml`。

## 2. 架构与路由

采用“路由驱动 + 按业务模块组织”：`app` 负责路由与组合，`features` 完整拥有业务功能，共享目录仅放跨业务能力。

```text
src/
├── app/
│   ├── (public)/                   # 首页、文章、分区、搜索、作者介绍
│   ├── (auth)/                     # 登录、注册
│   ├── author/                     # 作者后台，统一鉴权
│   ├── layout.tsx
│   ├── loading.tsx
│   ├── error.tsx
│   └── not-found.tsx
├── features/
│   └── <domain>/                   # api、components、hooks、schemas、types
├── components/                     # 跨业务 UI、布局、Markdown 组件
├── lib/                            # API 客户端、鉴权、Markdown、工具函数
├── hooks/                          # 跨业务 Hook
├── stores/                         # 必要的全局客户端状态
├── types/                          # 跨业务公共类型
└── config/                         # 站点和导航静态配置
```

### 2.1 依赖边界

- `app` 可依赖 `features`、`components`、`lib`；`features` 不得依赖 `app`。
- `components` 不得依赖具体业务模块。
- 业务专属请求、组件、Hook、Schema 和类型必须放在对应 `features/<domain>`。
- 仅被至少两个业务模块使用的能力才提升到共享目录。
- 禁止在顶层 `services`、`types` 与 `features` 重复维护同一业务代码。
- `stores` 不缓存服务端数据，仅用于确有必要的跨页面客户端状态。

### 2.2 路由与渲染

- 路由使用复数资源名；公开文章和分区 URL 使用可读的 `slug`。
- 文章编辑归入 `/author/articles`；作者后台在路由入口统一鉴权。
- 默认使用 Server Component；仅因浏览器 API、交互状态或客户端 Hook 添加 `"use client"`。
- 数据优先在服务端读取，只向客户端组件传递必要数据。
- `page.tsx`、`layout.tsx` 只负责参数、鉴权、数据装配和页面组合。
- 主要路由按需提供加载、错误、空数据、无权限和 404 状态。
- 未确认的评论、标签等功能不得提前实现或预留抽象。

### 2.3 权限

- 前端权限控制只负责展示与导航，后端必须独立校验权限。
- 未登录访问受限页面时跳转登录页；已登录但无权限时展示 `403`。
- 文章可见性使用后端计算结果，前端不重复推导继承规则。
- JWT 不得存入 `localStorage`，优先使用后端设置的 `HttpOnly`、`Secure` Cookie。

## 3. 数据与状态

- 所有请求通过统一 API 客户端发出，统一处理地址、请求头、超时、凭证和错误结构。
- 请求与响应必须有明确类型；外部数据进入界面前完成必要校验和转换。
- 服务端数据使用 Next.js 服务端请求与缓存能力管理。
- 临时交互状态保存在组件本地；确需跨页面共享时才创建 Store。
- 错误提示面向用户表达，不展示原始异常或后端堆栈。

## 4. 组件与文件拆分

- 组件保持单一职责；优先组合 HeroUI，项目级封装放入 `components/ui`。
- Props、函数参数和返回值必须有明确类型；命名应表达业务含义。
- 请求放入 `api/`，复杂状态与副作用放入 `hooks/`，校验与解析放入 `schemas/`，无状态计算提取为纯函数。
- 一个文件只承载一个主要组件或一组强相关能力。
- 注释解释原因和约束，不复述代码行为。

出现以下任一情况时，必须评估并按职责拆分：

- 文件同时承担请求、复杂状态、数据转换和大段渲染中的多个职责。
- JSX 存在两个及以上可独立命名和测试的业务区域。
- 存在重复逻辑，或一个组件包含多个无关表单、弹窗或流程。
- 修改局部功能前必须理解大量无关代码。
- 文件超过约 `300` 行，或单个组件超过约 `200` 行。

行数仅是检查信号。禁止为减少行数创建无业务含义、仅转发 Props、只使用一次且逻辑简单的组件、Hook 或文件；禁止为未知需求预留空目录和抽象。

## 5. 视觉、交互与内容

- 风格保持现代统一，使用统一设计令牌，优先保证阅读体验和内容层级。
- 页面必须适配移动端与桌面端，不得出现横向溢出、文字遮挡或不可操作控件。
- 交互控件必须具备默认、悬停、聚焦、禁用和加载状态。
- 表单必须有标签、校验提示、提交状态和失败恢复方式。
- 图标按钮提供可访问名称或 Tooltip；动效支持系统减少动态效果设置。
- Markdown 使用统一渲染组件和阅读样式，代码块统一由 Shiki 高亮。
- 用户内容必须安全过滤，禁止渲染未经处理的 HTML。
- 编辑器预览应尽量与发布后的文章展示一致。

## 6. 测试与完成标准

- 核心业务逻辑、权限判断和数据转换必须有单元测试。
- 登录注册、文章发布、可见性和用户审批等关键流程必须有集成或端到端测试。
- 修复缺陷时补充回归测试。

功能完成前，Agent 必须确认：

1. 行为、权限和架构边界正确，无明显重复或越层依赖。
2. 达到拆分信号的文件已评估并合理拆分。
3. 加载、空数据、错误、无权限及响应式状态完整。
4. 已在桌面端和移动端检查关键页面。
5. 已访问本次改动对应页面，确认页面可以正常打开且 Next.js 无报错。
6. 类型检查、Lint、测试和生产构建全部通过。
7. 架构或行为变化已同步更新文档。

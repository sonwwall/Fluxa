# 后端开发规范

本文档是 Fluxa 后端开发的执行基线。Agent 开发前必须阅读，并优先遵循现有代码与本规范。

## 1. 开发基线

- 使用 Go、Hertz、GORM、MySQL 构建单体后端服务。
- 使用 Viper 管理配置，使用 zap 作为统一日志系统。
- 采用 `handler -> service -> repository` 三层结构。
- 使用依赖注入组织模块依赖，禁止在业务代码中随意创建全局数据库连接或服务实例。
- 使用 `context.Context` 贯穿请求、业务逻辑和数据访问。
- 所有外部输入必须校验，所有权限必须由后端独立校验。
- 第一阶段使用单 JWT Token；双 Token 与 Redis 状态管理未确认前不得提前引入。

### 分层规则

- `handler`：解析请求、基础参数校验、调用 service、返回 HTTP 响应；不得直接访问数据库。
- `service`：业务编排、权限判断、状态流转、事务边界、领域规则；不得拼装 HTTP 响应。
- `repository`：数据库读写、查询条件封装、分页查询、事务内数据操作；不得承载权限判断、状态机规则或 HTTP 响应结构。
- `model` 不依赖 `handler`、`service` 或 `repository`。
- 跨业务调用优先通过对方 `service` 暴露的方法，不直接访问对方 `repository`。

典型调用链：

```text
HTTP -> middleware -> handler -> service -> repository -> MySQL
```

## 3. 模块边界与文件组织

首期模块包括 `article`、`category`、`user`、`profile`、`analytics`。未在准备文档确认的评论、标签等功能，不得提前开发或预留复杂抽象。

- 业务模块内部按职责目录组织，例如 `handler/`、`service/`、`repository/`、`model/`、`request/`、`response/`、`errs/`。
- 各职责目录内无论模块大小都应按业务意义拆分文件，例如 `list_public_articles.go`、`create_article.go`、`publish_article.go`、`approve_user.go`。
- `handler.go`、`service.go`、`repository.go` 只放类型定义、构造函数和该层共享的极少量基础能力，不承载多个业务流程。
- 每个核心功能应有对应测试文件，例如 `publish_article_test.go`。
- 复杂校验、状态流转、查询构造应提取为有业务含义的小函数。
- 命名必须表达业务意图，避免 `Handle`、`Process`、`Do` 等脱离上下文的泛化名称。
- 注释用于解释业务约束、边界条件和非显然决策，不复述代码行为。

出现以下情况时，应评估拆分：

- 单个文件同时包含多个无关业务流程或多种状态流转。
- handler 中出现复杂业务判断或事务逻辑。
- service 中出现大量 GORM 查询细节。
- repository 中出现权限判断、状态机规则或 HTTP 响应结构。
- 修改一个功能前必须理解大量无关代码。

## 4. 配置、安全与日志

- 使用 Viper 统一读取配置，禁止在业务代码中直接读取环境变量或配置文件。
- 配置读取集中在 `internal/config`，业务模块通过依赖注入接收配置值。
- `configs/config.yaml` 只放默认值和非敏感配置；密码、密钥等敏感配置通过环境变量覆盖。
- 启动时必须校验必要配置；缺少数据库、JWT、日志等关键配置时应启动失败。
- 业务日志使用 zap，禁止使用 `fmt.Println`、`log.Println` 打印业务日志。
- 以下场景必须记录结构化日志：服务启动与配置加载、鉴权或权限拒绝、关键状态变更、事务失败、外部依赖异常。
- 日志字段使用稳定命名，例如 `user_id`、`article_id`、`category_id`、`request_id`、`role`。
- 日志中不得打印明文密码、JWT、密钥、完整 Cookie 或其他敏感信息。
- 业务错误使用 `Warn`；系统异常、数据库异常、事务失败使用 `Error`；正常关键流程使用 `Info`。

## 5. API、错误码与响应

- API 路由使用版本前缀，例如 `/api/v1`。
- 路由面向资源命名，使用复数名词，例如 `/articles`、`/categories`、`/users`。
- 作者后台接口使用统一前缀，例如 `/api/v1/author`。
- 请求结构与响应结构分开定义，不直接暴露数据库模型。
- HTTP 响应体固定使用 `code`、`data`、`msg` 三个字段，禁止新增同级字段。
- 成功响应 `code` 固定为 `0`；业务错误必须有稳定错误码和用户可理解的错误信息。
- 底层错误不得直接返回给客户端；SQL、堆栈、依赖错误、JWT 解析细节等只能写入日志。

成功响应：

```json
{
  "code": 0,
  "data": {},
  "msg": "success"
}
```

分页响应：

```json
{
  "code": 0,
  "data": {
    "list": [],
    "page": 1,
    "page_size": 20,
    "total": 0
  },
  "msg": "success"
}
```

### 错误码规则

错误码使用 6 位数字：

```text
AABCCC
```

- `AA`：模块编号。
- `B`：错误所处层级。
- `CCC`：具体错误编号。

层级编号：

- `1`：handler/request 层，例如参数错误、请求格式错误。
- `2`：service/business 层，例如权限不足、状态不允许。
- `3`：repository/storage 层，例如数据不存在、唯一约束冲突。
- `9`：system/common 层，例如配置错误、外部依赖不可用。

模块编号：

- `00`：通用系统错误。
- `01`：auth。
- `02`：user。
- `03`：article。
- `04`：category。
- `05`：profile。
- `06`：analytics。

示例：

```go
var (
    ErrArticleNotEditable = errs.New(032001, "当前文章状态不允许编辑")
    ErrArticleNotFound    = errs.New(033001, "文章不存在")
)
```

每个业务模块必须提供 `errors.go`，集中维护本模块错误码、错误信息和错误变量。

## 6. 数据库、事务与权限

- 使用 GORM 访问 MySQL。
- 数据库模型放入对应业务包的 `model.go`，不要直接作为 HTTP 响应返回。
- 迁移脚本放入 `migrations/`，数据库结构变化必须同步迁移文件。
- 创建、发布、撤回、审批等涉及多表或状态一致性的操作必须使用事务。
- 事务由 service 层开启并控制边界，repository 提供事务内可复用的数据操作。
- 查询必须考虑分页、排序和必要索引。
- 软删除、唯一约束、时间字段和状态字段需要在模型与迁移中保持一致。
- 角色包括游客、注册用户、作者。
- 游客只能访问公开文章和公开作者信息；注册用户可访问公开内容和注册用户可见内容；作者拥有后台管理权限。
- 前端隐藏按钮不算权限控制，权限校验必须在后端完成。
- 作者接口必须经过统一鉴权中间件和 service 层权限校验。
- 文章可见性由后端统一计算：文章单独设置优先，否则继承分区可见性。

## 7. 测试要求

- 核心 handler、service、repository 方法应有对应测试。
- service 测试覆盖业务规则、权限判断、状态流转和异常路径。
- repository 测试覆盖关键查询、分页、事务和唯一约束冲突。
- handler 测试覆盖参数校验、鉴权结果、响应结构和错误映射。
- 错误码测试覆盖错误码唯一性、错误信息稳定性和底层错误不外泄。
- 修复缺陷时必须补充能复现该问题的回归测试。
- 提交前必须运行 `go test ./...`；无法运行时说明原因。

## 8. 提交规范

- 修改文件后默认不要提交 commit，也不要询问是否提交。
- 只有用户明确要求 `commit` 时，才读取 git 变动区并提交一次。
- commit 信息遵循 Conventional Commits，格式为 `<type>(scope): short description`，例如 `feat(auth): add login endpoint`。
- `type` 按实际变更选择，例如 `docs`、`feat`、`fix`、`refactor`、`test`、`chore`。
- commit 正文使用英文，并按条目分点描述具体修改。

## 9. 完成标准

功能完成前，Agent 必须确认：

- 代码符合三层职责和依赖方向，无越层调用。
- 权限、状态流转、参数校验和错误处理完整。
- 新增业务错误已定义错误码，客户端响应不泄漏底层信息。
- 关键节点已打印结构化日志，且不包含敏感信息。
- 数据库变更已同步迁移文件。
- 关键逻辑和新增功能已有对应测试。
- 架构或行为变化已同步更新文档。
